# Architecture

## High-Level Design
The system follows a pure Client-Side Rendering (CSR) architecture wrapped in a Progressive Web App (PWA). All "backend" logic is currently mocked and executed locally via IndexedDB.

### Layers

1. **View Layer (React Components)**
   - **Pages**: Top-level route containers (`src/pages`).
   - **Layouts**: Wrap pages to provide navigation, auth gating, and global providers (`src/components/layout`).
   - **UI Components**: Dumb, reusable presentational elements (`src/components/ui`).
   - **POS Components**: Domain-specific complex modules like `CartPanel` and `ProductGrid`.

2. **State Layer (Zustand)**
   - Decentralized stores for specific domain logic (Cart, Auth, Theme, Settings, Offline).
   - Some stores (Theme, Settings, Cart Status) use `localStorage` for immediate multi-tab synchronization or persistence.

3. **Data Layer (Dexie.js / IndexedDB)**
   - Acts as the primary source of truth.
   - `useLiveQuery` hooks connect components directly to database queries, enabling reactive UI updates when underlying data changes (e.g. stock decrements).

4. **Service Layer (`src/services`)**
   - **Hardware Proxies**: `printerService.ts`, `cashDrawerService.ts`. Native JavaScript construction of ESC/POS byte buffers instead of relying on external NPM packages.
   - **Barcode Utilities**: `barcodeService.ts` handles complex encoding/decoding for EAN-13, particularly weight-embedded barcodes generated by meat scales.

## Data Flow
1. User clicks "Pay" in `CartPanel`.
2. Cart items are formatted.
3. `Dexie.js` creates a new `Sale` record asynchronously.
4. `printerService.ts` generates a receipt.
5. `cartStore` is cleared.
6. A `StorageEvent` is broadcast to `CustomerDisplay` to reset the second-screen UI.
